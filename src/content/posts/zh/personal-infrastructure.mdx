## 先写母语，再做翻译，是更自然的路径

当你真正要长期写博客时，最昂贵的不是部署成本，而是表达成本。对大多数人来说，母语写作更快，也更能保持观点的细节和温度。

所以对双语博客来说，一个更稳的策略是：**中文先行，英文跟进**。

## AI 在这里更像“第二位编辑”

我不认为模型应该替代作者的判断，但它非常适合完成下面这些事：

- 基于中文原文生成英文初稿。
- 统一术语风格。
- 把冗长表达压缩得更自然。
- 帮你做标题、摘要和元信息改写。

这会让双语维护从“再写一篇”变成“再校一遍”。

## 一个简单可行的工作流

<aside className="mdx-note">
  你后续只需要把模型 API 配好，就可以在现有内容结构上继续自动化英文稿流程。
</aside>

1. 先在 `src/content/posts/zh` 写中文文章。
2. 保持相同的 `translationKey` 和 `pathSlug`。
3. 让 AI 生成英文初稿，落到 `src/content/posts/en`。
4. 你再做最后的人类校对和风格统一。

## 为什么这个结构适合 GitHub Pages

因为它仍然是纯文件工作流，完全免费，而且天然适合 Git 管理。博客内容像代码一样被版本化，长期可维护。
